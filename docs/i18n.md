# i18n Guide

## Directory Structure
- `src/locales/resources.ts`: runtime locale registry, supported locale list, and shared key type.
- `src/locales/index.ts`: single re-export entry for locale helpers.
- `src/locales/en/translations.json`: source of truth for the runtime key set.
- `src/locales/zh-CN/translations.json`: Simplified Chinese runtime locale.
- `src/locales/zh-TW/translations.json`: Traditional Chinese runtime locale.
- `src/locales/ja/translations.json`: Japanese runtime locale.
- `src/locales/zh/translations.json`: legacy file kept for compatibility/history only, not loaded by runtime resources.
- `src/i18n/*`: i18n bootstrap, locale detection, and singleton wiring.
- `src/popup/hooks/useTranslation.ts`: recommended app-facing translation hook.

## Entry Rule
- Business UI should use `useTranslation()` from `src/popup/hooks/useTranslation.ts`.
- Do not import `useTranslation` from `react-i18next` inside feature components.
- `react-i18next` stays in i18n bootstrap only.

## Key Naming
- Use dot-separated lowercase keys only.
- Allowed roots: `common.*`, `page.*`, `component.*`, `error.*`, `error.api.*`, `validation.*`.
- Keys must describe semantic purpose or business context, not English text.
- Do not use `text1`, `title2`, `btnText`, or similar placeholder keys.
- Do not reuse one key for different business scenarios because text looks similar.
- Put shared copy in `common.*`, page-specific copy in `page.*`, component copy in `component.*`, validation copy in `validation.*`, and API/runtime failures in `error.api.*`.

## Add Copy Flow
1. Add the semantic key to `en` first.
2. Add the same key to `zh-CN`, `zh-TW`, and `ja`.
3. Consume the key via `t()` from `@/popup/hooks/useTranslation`.
4. If a key is removed or renamed, update every locale file in the same change.
5. Run `corepack yarn compile`.
6. Run `corepack yarn build`.
7. Run `corepack yarn exec eslint src` when you need an i18n-only lint sweep.
8. Re-scan for hardcoded UI strings.

## Forbidden
- Do not change business logic.
- Do not change styles unless text length forces a layout fix.
- Do not add a new i18n library without explaining why the current stack is insufficient.
- Do not delete existing runtime locales.
- Do not let one locale drift from the others.
- Do not merge different business scenarios into one key.
- Do not introduce `any` in translation plumbing.
- Do not leave placeholder values like `???` or `????`.

## Examples
- `common.confirm`
- `common.cancel`
- `page.login.title`
- `page.login.password.placeholder`
- `component.header.language.select`
- `error.api.network`
- `validation.wallet.password.required`

## How to Find Hardcoded Text
- Search user-facing files under `src/popup`, `src/components`, `src/pages`, and `src/app`.
- Look for raw strings in titles, labels, placeholders, toast messages, modal text, empty states, loading states, tooltips, and aria labels.
- Skip debug logs, variable names, and non-UI literals.
- Check for `t('...')` calls still pointing at old keys.
- Use `rg -n '"[A-Za-z][^"]+"' src/popup src/components src/pages src/app --glob '!**/translations.json'` as a quick first pass.

## Keep Keys in Sync
- `en`, `zh-CN`, `zh-TW`, and `ja` must contain the same key set.
- Use `src/locales/validator.ts` to compare key coverage.
- Quick parity check command:

```bash
node -e "const fs=require('fs'); const l=['en','zh-CN','zh-TW','ja']; const d=Object.fromEntries(l.map(x=>[x,JSON.parse(fs.readFileSync('src/locales/'+x+'/translations.json','utf8'))])); const ref=Object.keys(d.en).sort(); for(const x of l){ const keys=Object.keys(d[x]).sort(); console.log(x, ref.length===keys.length && ref.every(k=>keys.includes(k)) ? 'ok' : 'mismatch'); }"
```

- After each copy batch, run `corepack yarn compile` and `corepack yarn build`.
